# CarbonCoach Numeric Invention Guard Specification

## Document Purpose

This document defines the **Numeric Invention Guard** for CarbonCoach.

The Numeric Invention Guard protects the application from LLM-generated unsupported numbers.

CarbonCoach uses Gemini for explanation and coaching, but all carbon calculations, impact bands, reduction estimates, tracker values, and progress values must come from deterministic TypeScript engines.

The guard ensures that Gemini does not accidentally introduce new carbon numbers that were not supplied by the deterministic system.

# 1. Core Rule

The central rule is:

```text id="vt5as2"
The LLM may explain numbers.
The LLM may not invent numbers.
```

If Gemini produces a number that is not present in the allowed deterministic context, the response must be rejected and replaced with a deterministic fallback response.

---

# 2. Why This Guard Exists

LLMs can produce plausible but unsupported numbers.

Example unsafe output:

```text id="kqxdoy"
Switching to metro could reduce your footprint by 40 kg CO2e this month.
```

If `40` was not generated by the deterministic recommendation engine, this is unsafe.

CarbonCoach must avoid:

- invented savings,
- invented percentages,
- invented distances,
- invented monthly totals,
- invented weekly totals,
- invented comparison ratios,
- invented rankings,
- invented progress values,
- invented exact precision.

The guard protects:

- user trust,
- security/responsibility score,
- testing score,
- manual review credibility,
- and product truthfulness.

# 3. Scope

## 3.1 Protected outputs

The guard must validate all LLM-generated text fields before returning a response to the frontend.

Protected response fields include:

```text id="o1o98y"
summary
explanation
recommendedNextStep
weeklyPlan[]
numbersUsed[]
confidenceNote
disclaimer
```

If additional text fields are added later, they must also be included.

## 3.2 Protected coach modes

The guard applies to:

```text id="1cvur3"
footprint_coach
choice_coach
```

Any future LLM mode must explicitly opt into numeric validation before release.

## 3.3 Guard execution location

The guard must run server-side in `services/api` before the response is returned to the frontend.

The shared implementation should live in:

```text id="77jtu5"
packages/shared/src/assistant/numericGuard.ts
```

The API should import and use that shared implementation.

# 4. Definitions

## 4.1 Allowed Number

An allowed number is a numeric value produced by deterministic CarbonCoach logic and explicitly included in the coach request context.

Examples:

```text id="p7uy1q"
216
82
64
48
12
2
```

Allowed numbers may come from:

- footprint total,
- category breakdown,
- recommended action reduction estimate,
- choice option numeric metadata,
- weekly plan numeric values,
- allowed impact estimate,
- explicitly permitted display value.

## 4.2 Generated Number

A generated number is any numeric value found in the LLM response text.

Examples:

```text id="5nmqtg"
40
3x
12%
216
7 days
```

Generated numbers must be checked against the allowed number set unless exempted.

## 4.3 Exempt Number

An exempt number is a number that may appear in natural language but is not a carbon or recommendation value.

Examples may include:

- list numbering,
- generic phrase “one small step” if written as word not digit,
- UI-neutral wording,
- ordinal text if not parsed as numeric.

To keep the first implementation simple, prefer avoiding digits in fallback and prompt-generated prose unless values are supplied.

# 5. Input Contract

Coach requests must include an allowed number list.

Recommended request shape:

```ts id="qf0bk0"
interface CoachRequest {
  mode: 'footprint_coach' | 'choice_coach';
  context: FootprintCoachContext | ChoiceCoachContext;
  allowedNumbers: string[];
}
```

## 5.1 Allowed numbers list

The `allowedNumbers` list must be derived from deterministic context.

Example:

```json id="d0yn5j"
{
  "allowedNumbers": ["216", "82", "64", "48", "12", "2"]
}
```

## 5.2 Allowed number normalization

Allowed numbers should be normalized before validation.

Recommended normalization:

```text id="5gyedk"
Trim whitespace
Remove commas
Normalize decimal formatting where practical
Store as canonical strings
```

Examples:

| Input   | Canonical                                     |
| ------- | --------------------------------------------- |
| `216`   | `216`                                         |
| `216.0` | `216` or `216.0`, depending on implementation |
| `1,200` | `1200`                                        |
| `12 kg` | `12`                                          |

Implementation must be consistent.

# 6. Output Contract

The LLM response should follow the shared coach response schema.

Recommended shape:

```ts id="4iuya6"
interface CoachResponse {
  mode: 'footprint_coach' | 'choice_coach';
  summary: string;
  explanation: string;
  recommendedNextStep: string;
  weeklyPlan: string[];
  numbersUsed: string[];
  confidenceNote: string;
  disclaimer: string;
  source: 'gemini' | 'fallback';
}
```

Before the response is returned, the guard must inspect all user-facing text fields.

# 7. Validation Algorithm

## 7.1 High-level algorithm

```text id="w44mrk"
1. Receive parsed CoachResponse.
2. Build allowed number set from request context.
3. Collect user-facing response text.
4. Extract numeric tokens from response text.
5. Normalize generated numeric tokens.
6. Compare generated tokens to allowed set.
7. If every generated number is allowed or exempt, accept.
8. If any unsupported number is found, reject.
9. On reject, return deterministic fallback response.
```

---

# 8. Numeric Extraction Rules

## 8.1 First implementation scope

The first implementation should detect common numeric formats:

```text id="nkyxua"
Whole numbers: 12
Decimals: 12.5
Percentages: 15%
Comma-separated numbers: 1,200
Simple multipliers: 3x
Negative numbers: -4
```

## 8.2 Suggested regex

A practical first-pass extractor may identify:

```regex id="bj9xyh"
-?\\d+(?:,\\d{3})*(?:\\.\\d+)?%?|\\d+(?:\\.\\d+)?x
```

Implementation may improve this if needed.

## 8.3 Numbers written as words

For P0, the guard does not need to convert words to numbers.

Examples:

```text id="rz5zyq"
two
three
twelve
```

However, prompts should instruct Gemini to avoid introducing unsupported numeric claims, including numbers written as words.

The product should prefer explicit deterministic numbers or qualitative wording.

# 11. Exemptions

## 11.1 Avoid exemptions where possible

To reduce complexity, avoid letting Gemini use unsupported digits at all.

Prompt should instruct:

```text id="c8wxd4"
If a numeric value is not provided, do not use a number.
Use qualitative wording instead.
```

## 11.2 Possible safe exemptions

The implementation may allow a limited set of safe numbers if they are part of fixed UI copy, not factual claims.

Examples:

```text id="d5iiit"
"one small step" written as word
"first" written as word
```

Avoid digit-based list numbering in LLM responses if it complicates validation.

Recommended:

- use arrays for `weeklyPlan`,
- render numbering in UI,
- do not ask LLM to include `1.`, `2.`, `3.` inside text.

---

# 12. Handling `numbersUsed`

The `numbersUsed` field must not be trusted blindly.

The guard should validate both:

1. numbers declared in `numbersUsed`,
2. numbers appearing in all other text fields.

If `numbersUsed` contains unsupported values, reject.

If text contains a number missing from `numbersUsed`, still validate against allowed numbers.

The `numbersUsed` field is helpful for transparency but not a substitute for extraction.

---

# 13. Fallback Behavior

If the guard rejects a response:

```text id="1dfcpk"
Do not return the unsafe LLM response.
Return deterministic fallback response.
Set source = "fallback".
Optionally include a small fallback note in the UI.
```

Fallback response must be generated from deterministic context and should not contain unsupported numbers.

# 18. Suggested Validation Behavior

## 18.1 Basic validation

```ts id="2exgxd"
const result = validateGeneratedNumbers({
  allowedNumbers,
  responseTextFields: collectCoachResponseTextFields(response),
});

if (!result.isValid) {
  return fallbackCoachResponse;
}
```

## 18.2 Normalization examples

Implementation should normalize:

| Raw token | Normalized                       |
| --------- | -------------------------------- |
| `12`      | `12`                             |
| `12%`     | `12` or `12%` based on strategy  |
| `1,200`   | `1200`                           |
| `3x`      | `3` or `3x` based on strategy    |
| `12.0`    | `12` or `12.0` based on strategy |

The implementation must document which strategy is used.

Recommended P0 strategy:

- store plain numeric value without `%` or `x`,
- treat `%` and `x` as numeric claim markers,
- require the base number to be allowed.

# 20. Edge Cases

## 20.1 Decimal precision

If deterministic context gives `12.5`, Gemini may output `12.50`.

The guard may normalize these as equivalent if implemented.

If not, prompt should instruct Gemini to use exact provided values.

## 20.2 Rounding

Do not allow Gemini to round values independently.

If rounded values are allowed, deterministic logic should include both raw and display values in allowed numbers.

Example:

```text id="jw3wyk"
allowedNumbers: ["12.5", "13"]
```

## 20.3 Ranges

If deterministic logic supplies a range, include both endpoints.

Example:

```text id="7db05n"
"10–15 kg" requires allowedNumbers: ["10", "15"]
```

## 20.4 Dates

Avoid dates in LLM output unless supplied.

If the weekly plan says “7 days,” include `7` if needed or avoid digit wording.

## 20.5 List numbering

The response should use arrays instead of embedded numbered lists.

The UI can render numbering separately.

This prevents generated `1`, `2`, `3` from being falsely treated as carbon values.

# 21. Prompt Guidance to Reduce Guard Failures

Prompts should say:

```text id="7eqxb0"
Do not create numbered lists inside text fields.
Use arrays for weeklyPlan items.
Do not introduce new percentages, kg values, distances, durations, costs, or multipliers.
If you need to compare choices without numbers, use impact bands.
Use only these allowed numbers: [allowedNumbers].
```

This reduces false rejects and improves user experience.

The Numeric Invention Guard is complete when:

# 24. Final Specification Statement

The Numeric Invention Guard is a required trust boundary in CarbonCoach.

It ensures Gemini can make the app feel intelligent and personalized without allowing the model to fabricate carbon values, savings, percentages, distances, or other numeric claims.

If in doubt, reject the LLM response and use deterministic fallback.
